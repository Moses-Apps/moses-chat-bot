# Admin Guide

For tenant admins and GlobalAdmins installing and operating moses-chat-bot.

## Installation via marketplace

1. In Moses, open **Admin** → **Marketplace** (or the per-tenant Apps view if
   you've enabled the marketplace there).
2. Find **Chat Bot Bridge** in the catalog. Verify the listing shows
   `name: moses-chat-bot`, `version: 0.1.0`, and the pinned source commit SHA.
3. Click **Install**. Moses clones the repo at the pinned commit into
   `moses-git/marketplace/moses-chat-bot/` for your tenant, runs the validator
   (the four commands in `moses-app.config.json#validation.commands`), builds
   both Docker images via Kaniko, and applies the Helm chart.
4. Wait for the deployment to go healthy in **Deployments** (typically 90-180s
   on a warm cluster). The bot exposes `/health` on both frontend (8080) and
   backend (8080).
5. The first time it boots, the backend creates four schema tables in its
   sidecar Postgres (provisioned by the Helm chart): `pending_links`,
   `chat_relay_links`, `chat_relay_messages`, `provider_chat_state`. Confirm
   the pod logs show `schema apply: ok`.

At this point the app is running but cannot relay anything — you still need
the Telegram bot token and the platform integration grant.

## BotFather setup

One Telegram bot per tenant.

1. In Telegram, open [@BotFather](https://t.me/BotFather).
2. Send `/newbot`. Pick a display name and a username ending in `_bot`
   (e.g. `moses_acme_bot`).
3. Copy the **HTTP API token** BotFather returns. It looks like
   `123456789:ABC-DEF1234ghIkl-zyx57W2v1u123ew11`.
4. Recommended bot settings via BotFather:
   - `/setprivacy` → **Enable**. The bot only sees commands directed at it and
     1:1 messages — it doesn't read group chat.
   - `/setjoingroups` → **Disable**. v1 is 1:1 only.
   - `/setdescription` → "Bridge to Moses Manager. Talk to your platform from
     anywhere."
   - `/setcommands` → paste the list from
     [USER_GUIDE.md § Slash commands](./USER_GUIDE.md#slash-commands).

Save the token. The next section installs it as a K8s Secret.

## Secrets

The bot reads four env vars at startup. Three come from K8s Secrets you create;
one is auto-generated by the Helm chart on first install.

| Env var | Source | Required | Rotation impact |
|---------|--------|----------|-----------------|
| `TELEGRAM_BOT_TOKEN` | K8s Secret you create from the BotFather token. | Yes (no inbound/outbound Telegram without it). | Replace the secret value, restart the bot pod. Existing links remain valid (they identify users by Telegram user ID, not token). |
| `TELEGRAM_WEBHOOK_SECRET` | K8s Secret (32-byte hex string you generate). Legacy bootstrap fallback only — the in-app **Connect Telegram** wizard generates and persists its own per-bot webhook secret. | No (long-polling, the default, uses no webhook secret). | Only relevant in opt-in webhook mode; replace the value and re-connect the bot so a fresh `setWebhook` runs. |
| `CHAT_BOT_MASTER_KEY` | Auto-generated by the Helm chart on first install, stored in a Secret tagged `helm.sh/resource-policy: keep`. | Yes. | **Critical**: rotating master invalidates ciphertexts encrypted under retired versions. Always **add** a new version alongside `active`, then sweep — never replace. See § Encryption at rest below. |
| `MOSES_PLATFORM_API_KEY` | Injected by Moses when the `moses-platform` integration grant is approved. | **Required**. Gates the externally reachable workspace-tool surface — see § Workspace-tool surface authentication below. The bot still relays inbound Telegram messages without it, but the `/api/v1/push/*` and `/api/v1/workspace/*` endpoints fail-closed (503) until the grant is approved. | Re-grant the integration to issue a fresh key; old key revoked automatically. |
| `BOT_PLATFORM_AUTH_DISABLED` | Set to `true` in a developer laptop to bypass the workspace-tool bearer gate. | **Never set outside dev.** The middleware logs a warn on every request when this is on; production deploys must leave it unset. | n/a — laptop-only escape hatch. |

Create the Telegram secret:

```bash
kubectl create secret generic telegram-bot-token \
  --namespace=<tenant-ns> \
  --from-literal=TELEGRAM_BOT_TOKEN=<botfather-token> \
  --from-literal=TELEGRAM_WEBHOOK_SECRET=$(openssl rand -hex 32)
```

Then add the secret name to the bot's Helm values (Moses' app-config UI exposes
this as the "Secret references" field):

```yaml
secrets:
  secretNames:
    - telegram-bot-token
```

`helm upgrade` after editing values; the new Secret mounts via envFrom on the
backend container.

## Granting the `moses-platform` integration

The bot declares this integration in `moses-app.config.json`:

```jsonc
"integrations": {
  "required": [
    {
      "type": "moses-platform",
      "key": "default",
      "scopes": ["chat_relay", "workspace_push"],
      "envMapping": {
        "apiKey": "MOSES_PLATFORM_API_KEY",
        "apiUrl": "MOSES_PLATFORM_API_URL"
      }
    }
  ]
}
```

To approve:

1. Moses UI → **Admin** → **Integrations** → find the bot's pending request.
2. Review the scopes (`chat_relay`, `workspace_push`) and approve.
3. Moses mints a platform key, mounts it as `MOSES_PLATFORM_API_KEY` on the
   backend pod, and restarts.

Until you approve:

- The bot still **relays inbound** Telegram → Moses Manager (each user's own
  API key carries that direction).
- Moses Manager **cannot push back** to users — the workspace-tool surface
  exists but MM has no key to call it with. Outbound push silently no-ops.
- External callers hitting `/api/v1/push/*` or `/api/v1/workspace/*` directly
  receive `503 platform_key_unset` (fail-closed; see next section).

## Workspace-tool surface authentication

The bot's workspace-tool endpoints (`/api/v1/push/*`, `/api/v1/workspace/*`)
are routed by the ingress and are therefore reachable from the public
internet. To prevent any external caller from spoofing
`X-Moses-Tenant-ID`, the bot wraps that mux in a bearer-token gate:

- The middleware `RequirePlatformAPIKey` constant-time-compares the inbound
  `Authorization: Bearer <token>` against `MOSES_PLATFORM_API_KEY`.
- If the env var is unset, every request fails-closed with
  `503 platform_key_unset`. This is intentional — silent acceptance of
  unauthenticated traffic was the bug we're closing.
- Only after this gate passes does the platform-injected
  `X-Moses-Tenant-ID` header become trustworthy; per-handler tenant binding
  checks still enforce defense-in-depth against tenant-crossing calls.

For local development without a platform pod, set
`BOT_PLATFORM_AUTH_DISABLED=true` in the bot's env. Every request is then
logged at WARN level so the misconfiguration is visible. **Never** set this
in any cluster you do not own.

## Inbound delivery mode

The bot receives Telegram messages by **long-polling** the `getUpdates` API by
default. This is purely outbound traffic: the bot pulls updates from Telegram,
so it needs **no public URL, no webhook, no HTTPS endpoint, and no tunnel**, and
it works behind the Moses per-app auth gate with no access-mode change. There is
nothing for you to configure — connecting a bot through the in-app **Connect
Telegram** wizard is sufficient.

The poll loop is owned by the bot backend: it `deleteWebhook`s any stale
webhook, then long-polls `getUpdates` and feeds each message into the same relay
path used by the webhook handler. The offset is held in memory; on a pod restart
the loop re-fetches recent updates, and the relay's `provider_message_id` dedup
guarantees no message is delivered twice.

> **Single replica**: `getUpdates` is single-consumer — exactly one poll loop
> may run per bot token. The chart ships `replicaCount: 1` for this reason.
> Scaling out would require webhook mode (below) or a leader-elected poller.

### Webhook mode (opt-in)

Webhook delivery is available as an opt-in for setups that want push delivery
and can expose a public HTTPS endpoint. Set `BOT_WEBHOOK_PUBLIC_URL` on the
backend to the scheme://host[/base-path] the bot is publicly reachable on:

```yaml
env:
  backend:
    BOT_WEBHOOK_PUBLIC_URL: "https://<moses-host>/apps/<tenant>/moses-chat-bot"
```

When this is set, connecting a bot registers a webhook at
`<BOT_WEBHOOK_PUBLIC_URL>/api/v1/providers/telegram/webhook` instead of starting
a poll loop. The bot generates a random `secret_token` (stored on the
`telegram_bot_config` row), passes it to `setWebhook`, and verifies the
`X-Telegram-Bot-Api-Secret-Token` header Telegram echoes on every delivery.
Leaving `BOT_WEBHOOK_PUBLIC_URL` unset keeps the default long-polling mode.

## Monitoring

- **Pod logs**: `kubectl logs -n <tenant-ns> -l app=moses-chat-bot-backend`. The
  bot logs JSON via `slog` at info level. Useful queries:
  - `slog.String("err", ...)` — every error path stamps `err`.
  - `"telegram poll: loop started"` / `"telegram poll: getUpdates failed; backing off"`
    — long-poll loop lifecycle and transient-error backoff.
  - `"telegram poll: bot token revoked; stopping loop"` — a revoked bot token
    (re-connect with a fresh token via the in-app wizard).
  - `"master_key_unavailable"` — a row's `encryption_key_id` references a master
    version not in the current Secret; user will be asked to re-link.
- **Message audit**: every relayed turn lands in `chat_relay_messages`. SQL-shell
  the bot's Postgres to spot-check delivery (joined to `chat_relay_links` for
  the Moses user).
- **Metrics**: v1 does **not** expose Prometheus metrics. Gap tracked for v1.1;
  for now, rely on logs and DB introspection.

## Encryption at rest

User API keys are stored ciphertext-only. The cipher is AES-256-GCM; the data
encryption key (DEK) is derived per tenant from a master key via HKDF-SHA256.

Master key Secret format (`CHAT_BOT_MASTER_KEY`, JSON-encoded):

```json
{
  "keys": {
    "v1": "<base64 of 32 random bytes>",
    "v2": "<base64 of 32 random bytes>"
  },
  "active": "v1"
}
```

`active` selects which version encrypts new writes. Older versions decrypt old
rows. The bot stamps each row's `encryption_key_id` with the version label
used.

### Rotation procedure

1. Generate a new 32-byte key: `openssl rand -base64 32`.
2. Edit the Secret to add `keys.v2 = "<new base64>"`. Do **not** remove `v1`.
3. Flip `active` to `v2`.
4. Send `SIGHUP` to the bot pod (or restart it). New ciphertexts now use `v2`.
5. A background sweeper re-encrypts existing rows from `v1` to `v2` over time
   (no user impact; users keep using the bot).
6. Once `kubectl exec` confirms zero rows with `encryption_key_id = 'v1'`, you
   may remove `v1` from the Secret. **Never remove a version while rows still
   reference it** — those users would be asked to re-link.

### Master loss recovery

There is no recovery. The K8s Secret is marked `helm.sh/resource-policy: keep`
so `helm uninstall` doesn't drop it, but if you delete the Secret manually
and lose the backup, every encrypted user key becomes unreadable and every
user re-links from scratch. **Back up the Secret** via your tenant's existing
GitOps / sealed-secrets flow.

## Capacity

- **Outbound rate**: 30 messages/minute per link (token bucket). The bot's
  `relay.Sender` enforces this to stay clear of Telegram's bot-API limits.
  Bursts above this queue; sustained excess returns 429 to the push API.
- **Inbound link-code mint**: 5/min per user (frontend → bot backend
  `/api/v1/links/codes`). Sized for "I fat-fingered, let me try again," not
  for automation.
- **Linking failures**: 3 wrong `/link <code>` attempts from one Telegram user
  → 15-minute lockout.
- **Per-link WS**: one persistent socket per linked user (the WS handshake
  embeds the user's MCP key, so connections can't be shared). The pool reaps
  idle sockets after 10 min via a 1-min sweeper.

What scales when:

- More **active users** → more WS sockets, more backend memory. Linear.
- More **inbound messages** → more `moses-backend` AI calls. The bot is rarely
  the bottleneck; Moses Manager is.
- More **outbound pushes** → token-bucketed per link; doesn't multiply with
  user count unless pushes target many users at once.

## Uninstalling

`helm uninstall` (or marketplace uninstall) drops the bot's Deployments,
Services, Ingress, and Postgres. **It does not drop**:

- The master-key Secret (`helm.sh/resource-policy: keep`). Manually
  `kubectl delete secret moses-chat-bot-master-key` if you want it gone —
  this is destructive (see § Master loss recovery).
- The Postgres PVC (Helm default). Decide whether to keep or `kubectl delete
  pvc -l app=moses-chat-bot-postgres`.
- Users' Moses API keys (`mcp_api_keys` rows live in the moses-platform
  database). Users keep their keys until they revoke from the Moses UI; the
  keys still work for direct `Authorization: Bearer` calls into Moses
  Manager, just not via this bot anymore.
- Telegram bot registration. The bot account in BotFather is untouched. If
  you reinstall later, you can reuse the same bot token. If you don't,
  `/deletebot` in BotFather to retire it.

Reinstalling on the same Postgres PVC re-uses old `chat_relay_links` rows — but
if you also rotated the master key in between, those ciphertexts are unreadable
and the affected users will be asked to re-link.

## Troubleshooting

**`TELEGRAM_BOT_TOKEN unset; telegram adapter not registered`** in backend logs

Expected when the Secret isn't mounted yet — the bot still serves push
endpoints (with "no providers" errors per link) and the linking UI. Configure
the Secret and restart.

**Webhook returns 401 from Telegram**

`TELEGRAM_WEBHOOK_SECRET` doesn't match. Re-run `setWebhook` with the value
currently in your Secret.

**Users see "Your Moses key was revoked"**

Their `chat_relay_links.deactivation_reason` will say `platform_401` (or
`master_key_unavailable` if you rotated incorrectly). Tell them to re-link.

**Push API returns `{ sent: false }` for every link**

No provider registered. Either `TELEGRAM_BOT_TOKEN` is unset, or the adapter
failed startup — check backend logs for `telegram adapter:` errors.

**Moses Manager isn't pushing anything**

`MOSES_PLATFORM_API_KEY` is missing — approve the integration grant in Moses
admin UI.
